# Em-Cubed Strategic Development Plan

## Executive Summary

Em-Cubed is an early-stage (v0.1.0) multi-surface skill framework enabling execution across Python, Prolog, and Hy Lisp surfaces. The project has a clear vision but suffers from code duplication, incomplete implementations, and minimal testing. This plan outlines a path to a stable, production-ready v1.0.

---

## 1. Codebase Architecture Assessment

### Current State
- **Dual Code Locations**: Code exists in both `src/em_cubed/` and root-level (`core/`, `surfaces/`, `api/`)
- **Package Structure**: Setuptools-based with `src/` layout
- **Multi-Surface Design**: Three execution surfaces (Python, Prolog/Hy) with unified indexing

### Architecture Diagram (Current)
```
em-cubed/
├── src/em_cubed/          # Package (indexer, search, surfaces/)
├── core/                   # Duplicate registry/indexer (unused)
├── surfaces/               # Duplicate surface adapters (unused)
├── api/                    # FastAPI layer (stub endpoints)
├── skills/                 # Skill definitions (manifest.yaml minimal)
├── examples/               # POC multi-surface skill
├── tests/                  # Minimal test coverage
└── docs/                   # MULTI_SURFACE.md spec
```

### Issues Identified
1. **Code Duplication**: `src/em_cubed/surfaces/` vs `surfaces/` - same abstractions duplicated
2. **Unused Components**: `core/indexer.py` and `core/registry.py` are not integrated
3. **Broken Imports**: Tests import from `surfaces.*` and `core.*` instead of `em_cubed.*`
4. **API-Search Disconnect**: API endpoints are stubs, not connected to search/indexer

## 4. Technical Debt & Code Quality

### High Priority Debt
1. **Code Duplication**
   - `src/em_cubed/surfaces/` vs `surfaces/` - Remove root-level duplicates
   - `src/em_cubed/indexer.py` vs `core/indexer.py` - Consolidate to package

2. **Broken Test Imports**
   ```python
   # Current (broken):
   from core.indexer import Indexer
   from surfaces.python_runner import PythonRunner
   
   # Should be:
   from em_cubed.indexer import reindex
   from em_cubed.surfaces import PythonSurface
   ```

3. **Security Issue - Python eval()**
   - `python_runner.py` uses `eval()` with minimal sandboxing
   - `__builtins__: {}` is not secure (can escape via `__import__`)
   - Recommend: Use `asteval` or `restrictedpython` library

4. **Dependency Management**
   - Dual dependency files cause confusion
   - Consolidate to pyproject.toml only

### Medium Priority Debt
1. **Type Hints Missing**
   - Many functions lack return type annotations
   - Surface.execute signature could be more specific

2. **Error Handling**
   - Bare exceptions (`except Exception`) throughout
   - No structured error responses from API

3. **Configuration Management**
   - No config file for paths, settings
   - Hardcoded assumptions about directory structure

## 8. Security Considerations

### Current Issues
1. **Python eval() in `python_runner.py`**
   - Minimal sandbox can be bypassed
   - No timeout on execution
   - No resource limits

2. **Prolog Integration**
   - pyswip executes arbitrary Prolog - needs sandboxing
   - SWI-Prolog installation requirement not documented

3. **API Security**
   - No authentication on endpoints
   - No rate limiting
   - No input validation (beyond Pydantic)

4. **File System Access**
   - Indexer reads arbitrary SKILL.md files
   - No path traversal protection

### Recommendations
- Replace eval() with `asteval` or `restrictedpython`
- Add execution timeouts for all surfaces
- Add API authentication (JWT or API keys)
- Validate and sanitize all file paths
- Add security policy document

---

## 9. Scalability Constraints

### Current Limitations
1. **In-Memory Registry**
   - `search.py` loads entire registry.json into memory per query
   - Not suitable for large skill collections

2. **Whoosh Integration Missing**
   - `requirements.txt` includes Whoosh but not used
   - Could provide proper indexing for larger datasets

3. **No Caching**
   - Skills re-indexed on each call
   - No caching of surface execution results

4. **Single-Process API**
   - No worker model for concurrent skill execution
   - Long-running Prolog/Hy tasks block the event loop

### Scalability Path
- Implement Whoosh for persistent indexed search
- Add Redis for caching (optional dependency)
- Support async surface execution with task queue (Celery or similar)
- Add pagination to search results

## 13. Recommended Workflows

### Branching Strategy (GitFlow-inspired)
```
main (production)
│
├── develop (integration)
│   ├── feature/skill-cli
│   ├── feature/prolog-adapter
│   └── bugfix/indexer-paths
│
├── release/v0.2.0
└── hotfix/security-patch
```

**Rules:**
- `main`: Always deployable, protected branch
- `develop`: Integration branch, require PR reviews
- `feature/*`: Short-lived, merge to develop
- `release/*`: Prepare releases, bug fixes only
- `hotfix/*`: Critical fixes to main, merge to main + develop

### Commit Convention (Conventional Commits)
```
feat: add Prolog adapter implementation
fix: resolve import path issues in test_surfaces.py
docs: update README with quick start guide
refactor: consolidate duplicate surface code
test: add integration tests for search module
```

### PR Process
1. Create feature branch from develop
2. Implement changes with tests
3. Run `ruff check`, `mypy`, `pytest` locally
4. Create PR to develop with description template
5. Require 1-2 reviewer approvals
6. All CI checks must pass
7. Squash merge to keep history clean

## 14. Code Quality Standards

### Linting & Formatting
- **Ruff**: All code must pass `ruff check .`
- **Line Length**: 120 characters (configured in pyproject.toml)
- **Python Version**: Target 3.11+ (type hints, match statements)

### Type Checking
- **MyPy**: Strict mode enabled
- All public functions must have type hints
- Use `from __future__ import annotations` for forward references

### Testing Standards
- **Coverage**: 80%+ for core modules, 90%+ before v1.0
- **Naming**: `test_<module>_<function>_<scenario>`
- **Structure**: Arrange-Act-Assert pattern
- **Mocking**: Mock external services, use fixtures for common setup

### Documentation
- All public APIs must have docstrings
- Use Google style or NumPy style docstrings
- Keep README.md up to date with latest features
- Architecture decisions documented in `docs/adr/`

## 15. Governance Practices

## 17. Risk Mitigation Tactics

### Identified Risks

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Prolog integration complexity | High | High | Start with pyswip, document SWI-Prolog deps clearly |
| Security vulnerability in eval() | High | High | P0: Replace with asteval immediately |
| Dependency conflicts | Medium | Medium | Single source of truth (pyproject.toml) |
| Breaking API changes | High | Medium | Version endpoints, deprecation warnings |
| Test coverage gaps | Medium | High | Enforce coverage gates in CI |
| Skill ecosystem adoption | High | Medium | Create compelling example skills |

### Contingency Plans
1. **If Prolog adapter proves too complex**:
   - Pivot to Janus-Prolog (different library)
   - Or defer Prolog to v0.3.0, focus on Python/Hy first

2. **If security audit fails**:
   - Immediate security-focused release (v0.1.1)
   - Disable Python surface until fixed

3. **If test coverage stalls**:
   - Reduce scope for v0.2.0
   - Pair programming sessions for test writing

## 20. Conclusion

Em-Cubed has a solid conceptual foundation with its multi-surface skill framework. The primary focus for the next 3 months should be:

1. **Eliminate technical debt** (code duplication, broken imports)
2. **Complete core functionality** (Prolog adapter, API integration)
3. **Establish quality standards** (testing, CI/CD, linting)
4. **Build example ecosystem** (demonstrate multi-surface patterns)

By following this plan, the project can progress from a promising prototype (v0.1.0) to a stable, usable framework (v0.2.0) and eventually a production-ready system (v1.0).
